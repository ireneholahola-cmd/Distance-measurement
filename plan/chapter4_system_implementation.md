### 4.1.2 系统整体流程

第3章将技术方案划分为"数据输入与感知检测→风险量化→人机交互"三个阶段。在工程实现中，这三个阶段被映射为一条严格串行的数据处理流水线，每一阶段的输出即为下一阶段的输入，形成逻辑闭环。系统整体流程如图4-1所示。

```
┌─────────────────────────────────────────────────────────────────────┐
│                    阶段一：数据输入与感知检测                          │
│                                                                     │
│  视频输入 ──→ 预处理 ──→ YOLOv10检测 ──→ DeepSORT跟踪               │
│      │                                    ↑                         │
│      └──────→ Depth Anything V2 ──→ 3D重建+滤波 ──┘                 │
│                                                                     │
│  输出: {track_id, class, (X,Y,Z), (W,H,L), ψ, (vx,vy,vz)}         │
├─────────────────────────────────────────────────────────────────────┤
│                    阶段二：风险量化                                   │
│                                                                     │
│  3D轨迹 ──→ 高斯风险场构建 ──→ SCF计算 ──→ 风险分级                 │
│               ↑                                                     │
│  路面检测 ──→ 路面风险场 ──→ 动态+路面融合 ──┘                       │
│                                                                     │
│  输出: {SCF值, 风险分布图, 风险等级(HIGH/MEDIUM/LOW/CLEAR)}         │
├─────────────────────────────────────────────────────────────────────┤
│                    阶段三：人机交互                                   │
│                                                                     │
│  风险数据 ──→ BEV鸟瞰图渲染 ──→ Streamlit UI呈现 ──→ 驾驶员        │
│            ──→ 分级警报生成 ──→ 实时推送 ──────────┘                 │
└─────────────────────────────────────────────────────────────────────┘
```

**图4-1 系统整体流程图**

该流程的核心特征是**严格的数据依赖关系**：阶段二的输入完全由阶段一决定，阶段三的输入完全由阶段二决定。这种串行依赖意味着：**阶段一的精度上限决定了整个系统的精度上限**。例如，若目标检测漏检一辆车，则该车的深度、3D位置、风险值均无法计算，整条链路上该目标彻底消失。这一"精度传递"特性决定了系统优化的优先级——感知阶段的精度提升对最终效果的边际收益远大于后续阶段，因此在工程实现中，感知模块的参数调优和算法选择获得了最高的优先级。

值得注意的是，图4-1中存在一条关键的**数据反馈路径**：3D重建的结果（3D位姿）被回传至DeepSORT跟踪器，用于增强帧间关联的3D欧氏距离匹配。这一反馈设计打破了严格的串行流水线结构，形成了"检测→3D重建→跟踪→3D重建"的迭代优化环路。该环路的存在使得3D重建的精度不仅影响风险量化，还反过来影响跟踪的准确性——3D位置越精确，跟踪的帧间关联越可靠，从而进一步提升后续帧的3D重建质量。这种正反馈机制是本系统在标准DeepSORT基础上的核心改进之一。

## 4.2 软件模块设计与实现

### 4.2.1 视频输入与预处理模块

#### 功能定位

视频输入与预处理模块是整条流水线的起点，负责将原始视频流转换为后续模块可消费的标准格式。该模块的核心设计原则是**统一接口**——无论输入源是视频文件、摄像头还是网络流，后续模块看到的都是统一尺寸、统一格式的帧数据。这一抽象层的存在使得上游输入源的切换对下游模块完全透明，是系统可扩展性的基础保障。从软件工程的角度看，预处理模块扮演了"适配器"（Adapter）的角色，将异构的输入源适配为统一的输出接口，符合"依赖倒置原则"——下游模块依赖抽象的帧接口，而非具体的输入源实现。

#### 输入源适配

系统支持三种输入源，通过`--source`参数切换：

| 输入源 | 参数格式 | 实现方式 | 典型场景 |
|--------|---------|---------|---------|
| 视频文件 | `path/to/video.mp4` | `cv2.VideoCapture(path)` | 离线分析、算法验证 |
| 摄像头 | `0`（设备索引） | `cv2.VideoCapture(0)` | 实时车载场景 |
| 图片序列 | `path/to/images/` | 逐帧读取 | 数据集评估 |

对于网络流输入（如RTSP协议的IP摄像头），系统采用独立线程异步读取帧数据，并设置缓冲区大小为3帧（`cv2.CAP_PROP_BUFFERSIZE = 3`），以平衡实时性与帧丢失风险。当处理速度跟不上输入帧率时，系统自动丢弃旧帧，确保始终处理最新的画面——在主动安全预警场景中，"看到最新画面"比"处理每一帧"更重要，因为过时的感知信息可能导致错误的预警判断。

#### 预处理流水线

每一帧图像在进入检测模块前，需经过以下预处理步骤。该流水线的设计目标是：将异构的输入帧转换为YOLOv10和Depth Anything V2各自要求的标准化输入格式，同时保留原始帧用于可视化。

**1. 尺寸标准化（Letterbox缩放）**

YOLOv10要求输入尺寸为32的倍数（因网络中存在5次2倍下采样，$2^5=32$）。系统将原始图像缩放至目标尺寸（默认640×640），同时记录缩放比例用于后续将检测坐标映射回原始图像：

$$\text{scale} = \min\left(\frac{W_{target}}{W_{orig}}, \frac{H_{target}}{H_{orig}}\right)$$

缩放时保持宽高比，不足部分用灰色（像素值114，即ImageNet数据集的均值像素）填充。这一Letterbox策略相比直接Resize的优势在于：不引入图像畸变，检测框的映射仅需简单的缩放+偏移即可完成，避免了双线性插值带来的亚像素误差。填充像素值选择114而非0或255的原因是：114接近ImageNet训练集的均值亮度，对YOLOv10的卷积激活影响最小，不会在填充区域产生虚假的边缘响应。

**2. 色彩空间转换**

YOLOv10要求RGB输入，而OpenCV默认读取BGR格式。转换关系为：

$$I_{RGB} = \text{cv2.cvtColor}(I_{BGR}, \text{cv2.COLOR\_BGR2RGB})$$

**3. 归一化**

将像素值从[0, 255]归一化到[0, 1]：

$$I_{norm} = \frac{I}{255.0}$$

YOLOv10的训练数据采用了相同的归一化策略，推理时必须保持一致，否则将导致精度严重下降。这一归一化的数学本质是将输入映射到激活函数（如SiLU）的敏感区间，确保梯度信息能够有效传播。

**4. 张量化**

将HWC格式转换为NCHW格式，并增加batch维度：

$$I_{batch} = I_{norm}^{HWC} \xrightarrow{\text{transpose}} I_{norm}^{CHW} \xrightarrow{\text{unsqueeze}} I_{batch}^{1 \times C \times H \times W}$$

这一转换的必要性在于：PyTorch的卷积层要求输入为NCHW格式（Batch×Channel×Height×Width），而NumPy/OpenCV的默认格式为HWC。`unsqueeze`操作增加的batch维度使得单帧图像也能以统一的方式送入模型，与批量推理的代码路径完全一致。

#### 模块输出

预处理模块的输出被分流至两个下游消费者，形成"一帧两用"的数据分发模式，如表4-3所示。

**表4-3 预处理模块输出及下游消费者**

| 输出项 | 数据格式 | 下游消费者 | 消费方式 |
|--------|---------|-----------|---------|
| 原始帧 | numpy array (H×W×3, BGR) | Depth Anything V2 | BGR→RGB→PIL转换后输入深度估计pipeline |
| 原始帧 | numpy array (H×W×3, BGR) | 可视化模块 | 在原始帧上绘制检测框、3D框、风险投影 |
| 预处理帧 | tensor (1×3×640×640, RGB) | YOLOv10检测 | 直接送入模型前向推理 |
| 帧元信息 | {frame_idx, total_frames, fps} | 进度显示、速度计算 | 帧索引用于进度条，fps用于速度估算 |

值得注意的是，原始帧被同时传递给深度估计模块和可视化模块，而预处理帧仅传递给YOLOv10。这一分流策略的原因是：Depth Anything V2的HuggingFace pipeline接口接受PIL格式输入，其内部自行完成尺寸调整和归一化，因此无需使用YOLOv10的预处理结果。这种"一帧两路"的处理模式虽然增加了内存开销（同时保存原始帧和预处理帧），但避免了深度估计模块重复执行预处理步骤，从整体效率上看是更优的选择。在内存受限的嵌入式场景中，可通过共享内存或零拷贝技术进一步优化内存占用。

### 4.2.2 目标检测与跟踪模块

#### 功能定位

本模块实现第3章"数据输入与感知检测阶段"的核心功能——从视频帧中检测交通参与者并建立帧间关联。模块由两个子模块串联组成：YOLOv10目标检测（逐帧独立）和DeepSORT多目标跟踪（帧间关联）。两者的协作关系并非简单的串行——3D重建模块的输出（3D位姿）被回传至DeepSORT，用于增强帧间关联的3D欧氏距离匹配，形成"检测→3D重建→跟踪→3D重建"的迭代优化环路。这一反馈环路的设计动机是：标准DeepSORT仅使用2D信息（外观特征+IoU）进行匹配，在同类车辆密集场景中容易产生ID切换；而3D空间位置在帧间具有强连续性，可作为有效的辅助匹配信号。

#### 4.2.2.1 YOLOv10目标检测实现

**模型加载与推理**

YOLOv10-S模型（7.2M参数）通过Ultralytics框架加载，推理流程如下：

1. **模型加载**：加载预训练权重`yolov10s.pt`，根据`--device`参数分配计算设备
2. **模型预热**：在正式推理前，送入一个零张量进行前向传播，触发JIT编译和显存预分配，避免首帧延迟
3. **前向推理**：将预处理后的张量送入模型，依次经过Backbone（特征提取）、Neck（特征融合）、Head（预测头）三个阶段
4. **后处理**：YOLOv10无需NMS（这是选择它的核心原因），直接输出最终检测结果
5. **类别过滤**：仅保留与交通安全相关的5个类别

**YOLOv10无需NMS的技术原理**

YOLOv10相较于YOLOv8的核心改进在于**消除NMS（非极大值抑制）**。传统YOLO系列在推理后需要NMS去除重复检测框，NMS的IoU阈值选择对精度和召回率有显著影响，且NMS本身是不可并行化的串行操作，在GPU上成为延迟瓶颈。YOLOv10通过引入**一致双重分配策略（Consistent Dual Assignment）** [[1]](https://arxiv.org/abs/2405.14458)，在训练阶段同时使用一对多和一对一的标签分配策略，推理时仅使用一对一分支的输出，从而天然无需NMS。具体而言，一对多分支负责提供丰富的监督信号以学习高质量的特征表示，一对一分支则确保每个目标只产生一个预测框，两者在训练时通过损失加权保持一致性。这一设计使YOLOv10-S的端到端推理延迟降低至约8ms（GPU），相比YOLOv8-S的约12ms减少了33%，且消除了NMS阈值这一敏感超参数的调优负担。

**检测输出的数据结构：**

每个检测目标表示为一个7元组：

$$\text{detection} = (x_{min}, y_{min}, x_{max}, y_{max}, \text{conf}, \text{class\_id}, \text{class\_name})$$

其中 $(x_{min}, y_{min}, x_{max}, y_{max})$ 为2D边界框的像素坐标（已通过`scale_coords`函数映射回原始图像尺寸），$\text{conf}$ 为置信度分数，$\text{class\_id}$ 为COCO类别ID。

**类别过滤机制：**

COCO数据集包含80个类别，本系统仅保留与交通安全相关的5个类别，过滤逻辑如下：

$$\text{valid} = \begin{cases} \text{True} & \text{class\_id} \in \{1, 2, 3, 5, 7\} \\ \text{False} & \text{otherwise} \end{cases}$$

**表4-4 保留类别及物理含义**

| 保留类别ID | 类别名称 | 物理含义 | 先验3D尺寸(H×W×L, m) |
|-----------|---------|---------|---------------------|
| 1 | bicycle | 自行车/非机动车 | 1.40 × 0.70 × 1.80 |
| 2 | car | 小汽车 | 1.52 × 1.64 × 3.85 |
| 3 | motorcycle | 摩托车 | 1.50 × 0.90 × 2.20 |
| 5 | bus | 公交车/大客车 | 3.07 × 2.63 × 11.17 |
| 7 | truck | 卡车/载货车 | 3.07 × 2.63 × 11.17 |

类别过滤的设计考量是：在驾驶场景中，行人（class_id=0）虽然也是重要的交通参与者，但由于行人的3D尺寸估算误差极大（行人姿态变化大、遮挡严重），且行人速度通常较低，在当前版本中暂不纳入风险量化流程。未来版本可通过引入行人专用3D模型（如3DMPPE [[4]](https://arxiv.org/abs/1904.04475)）来扩展此功能。此外，COCO中的"stop sign"（class_id=11）、"parking meter"（class_id=12）等交通设施类别虽与驾驶相关，但属于静态目标，不产生动态风险场，因此也被过滤。

**置信度阈值调节：**

系统提供可调置信度阈值（默认0.25）。该阈值的设计权衡是：

- 阈值过低 → 误检增多（后续模块可过滤），但漏检减少（安全性提高）
- 阈值过高 → 误检减少，但漏检增多（安全性降低）

本系统采用"宁可误检不可漏检"的策略，默认阈值设为较低的0.25，因为**漏检意味着该目标在整条链路上彻底消失，后果远比误检严重**。误检目标可通过后续的3D重建合理性检查（如尺寸是否在合理范围内、深度值是否有效）和跟踪器的置信度衰减机制自然淘汰——误检目标通常不会在连续帧的同一位置重复出现，DeepSORT的试探确认机制（需连续3帧命中）可有效过滤此类噪声。

#### 4.2.2.2 DeepSORT多目标跟踪实现

**为什么检测之后还需要跟踪？**

YOLOv10在每一帧独立检测，产出一组无序的2D边界框。但风险量化需要两个关键信息——**速度**和**轨迹**——它们只能通过帧间关联获得：

- 速度 = 同一目标在相邻帧的3D位移 / 帧间隔
- 轨迹 = 同一目标在连续帧的位置序列

没有跟踪，速度和轨迹都无法计算，风险场中的速度拉伸效应将无法体现，轨迹预测模块也将失去输入。因此，跟踪模块是从"静态检测"到"动态分析"的关键桥梁，它赋予了系统时间维度上的感知能力。

**DeepSORT的三重匹配机制**

本系统在标准DeepSORT基础上增加了3D欧氏距离匹配，形成三重匹配机制：

$$c_{ij} = \lambda_1 \cdot d_{appearance}(i,j) + \lambda_2 \cdot d_{IoU}(i,j) + \lambda_3 \cdot d_{3D}(i,j)$$

三种度量的互补关系如表4-5所示。

**表4-5 三重匹配度量的互补关系**

| 度量 | 计算方式 | 擅长场景 | 失效场景 | 权重 $\lambda$ |
|------|---------|---------|---------|-------------|
| 外观特征 $d_{appearance}$ | 128维ReID向量的余弦距离 | 目标外观独特（不同颜色/类型） | 同类目标外观相似（同色车队） | 0.4 |
| 2D IoU $d_{IoU}$ | 边界框交并比 | 目标2D位置连续变化 | 遮挡后重现、快速运动 | 0.3 |
| 3D距离 $d_{3D}$ | 欧氏距离 $\sqrt{\Delta X^2+\Delta Y^2+\Delta Z^2}$ | 目标3D位置稳定 | 3D重建噪声大 | 0.3 |

三重匹配的核心价值在于**鲁棒性冗余**：当某一度量失效时（如遮挡导致IoU为0），其他度量仍可正确关联，避免ID切换。具体而言：

- **同色车队场景**：外观特征失效（同色车辆ReID向量高度相似），但IoU和3D距离仍可区分——不同车辆在2D和3D空间中占据不同位置
- **遮挡后重现场景**：IoU失效（遮挡期间无2D框），但外观特征可通过ReID重新识别，3D距离可基于运动预测提供辅助
- **快速横穿场景**：IoU失效（帧间位移过大导致框无重叠），运动模糊也影响ReID特征，但3D位置仍可通过匀速运动模型预测

**DeepSORT的匹配流程**

DeepSORT采用**级联匹配+IoU匹配**的两阶段匹配策略。级联匹配优先匹配低age（最近被更新过）的轨迹，确保高优先级的活跃轨迹优先获得匹配，避免新出现的误检目标"抢占"已有轨迹的匹配权。IoU匹配作为补充，处理级联匹配未匹配的高置信度检测。这种两阶段策略在保证匹配准确性的同时，有效降低了ID切换率。

**轨迹确认机制**

DeepSORT采用"试探→确认"的两阶段轨迹管理策略。新检测到的目标首先进入试探状态（Tentative），只有连续命中 $n_{init}=3$ 帧后才被确认为有效轨迹（Confirmed）。这一机制有效过滤了误检目标——误检通常不会在连续3帧的同一位置重复出现，而真实目标则具有时空连续性。确认后的轨迹在丢失后仍可保持 $max\_age=30$ 帧的存活期，期间若目标重新出现可恢复跟踪，避免短暂遮挡导致的ID切换。$max\_age=30$ 的设置对应约1秒的容忍时间（以30fps计算），足以应对大多数短暂遮挡场景（如被前方车辆短暂遮挡）。

**速度估算实现**

速度基于帧间3D位移计算：

$$v = \frac{\sqrt{(X_t - X_{t-1})^2 + (Z_t - Z_{t-1})^2}}{\Delta t} \times 3.6 \quad (\text{km/h})$$

其中 $\Delta t = 1/\text{fps}$ 为帧间隔。该公式的精度取决于3D位置的准确性，因此3D重建模块的双重滤波对速度估算的稳定性至关重要。未经滤波的3D位置帧间抖动可达1-2米，以30fps计算，这将引入约108-216 km/h的速度噪声，完全不可用。经过双重滤波后，3D位置抖动降至0.1-0.2米，速度噪声降至约11-22 km/h，虽仍有波动但已具备参考价值。为进一步提升速度估算的稳定性，系统对速度值也进行了EMA平滑处理，平滑后的速度变化率被限制在物理合理的范围内（加速度不超过 $0.5g \approx 5\text{m/s}^2$），避免了异常速度跳变对风险场计算的影响。

**模块输出**

| 输出项 | 数据格式 | 后续消费者 | 关键用途 |
|--------|---------|-----------|---------|
| 跟踪ID | 整数 | 风险场引擎 | 区分不同目标的风险源 |
| 2D框+类别 | 同检测输出 | 3D重建模块 | 提供2D定位和类别先验 |
| 速度 | (vx, vz) m/s | 风险场引擎 | 速度拉伸效应的方向和幅度 |
| 轨迹历史 | [(X,Z,t), ...] | BEV可视化 | 轨迹渐隐线绘制 |
| 3D位姿 | (X,Y,Z,W,H,L,ψ) | 风险场引擎 | 风险源的空间位置和尺寸 |

至此，感知检测阶段的核心模块——目标检测与跟踪——已完整实现。从原始帧到跟踪输出，数据经历了"2D检测→3D重建→帧间关联"的完整转换链。然而，跟踪模块输出的3D位姿仍存在帧间抖动问题，这直接影响了速度估算和风险场计算的稳定性。下一节将深入3D重建模块的实现细节，特别是双重滤波机制如何解决这一核心工程难题。

### 4.2.3 深度估计与3D重建模块

#### 功能定位

本模块解决第3章提出的核心问题——**从2D图像恢复3D空间信息**。这是整个系统从"检测"升级到"空间感知"的关键跃迁，也是技术难度最高的模块。模块由两个子模块组成：Depth Anything V2深度估计和3D边界框重建。前者的任务是将2D像素的"深度缺失"问题转化为可计算的深度值，后者的任务是将深度值与2D检测框结合，通过几何约束恢复目标的3D空间位置和尺寸。两个子模块的串联关系体现了"先感知深度、再重建空间"的工程逻辑——没有可靠的深度信息，3D重建无从谈起；而仅有深度信息，不经过几何约束和滤波处理，3D重建的结果也不具备实用性。

#### 4.2.3.1 Depth Anything V2深度估计实现

**模型加载策略**

Depth Anything V2通过HuggingFace的pipeline接口加载，支持三种模型尺寸。本系统选择Small模型（25M参数），其推理延迟在CPU上约100ms/帧，是唯一满足实时性要求的选择。模型尺寸的对比如表4-6所示。

**表4-6 Depth Anything V2模型尺寸对比**

| 模型 | 参数量 | CPU延迟 | GPU延迟 | 相对精度 | 本系统是否采用 |
|------|--------|---------|---------|---------|-------------|
| Small | 25M | ~100ms | ~11ms | 基准 | ✅ 采用 |
| Base | 97M | ~350ms | ~30ms | +4% δ1 | ❌ 延迟过高 |
| Large | 335M | ~800ms | ~60ms | +7% δ1 | ❌ 延迟过高 |

其中 $\delta_1$ 为深度估计的精度指标，表示预测深度与真实深度的比值在 $[1.25^{-1}, 1.25]$ 范围内的像素百分比。Base和Large模型虽精度更高，但其CPU延迟远超实时性要求（>200ms/帧），在无GPU环境下不可用。即使在GPU环境下，Large模型的60ms延迟加上YOLOv10的5ms延迟，总延迟已达65ms，对应约15FPS的帧率，在快速运动场景中可能产生明显的感知延迟。

**设备自适应策略：**

系统实现了自动设备分配，解决YOLOv10与Depth Anything V2的显存竞争问题：

| 检测到的设备 | YOLOv10设备 | Depth Anything设备 | 理由 |
|-------------|-----------|------------------|------|
| CUDA GPU (≥6GB) | GPU | CPU | 避免显存溢出，YOLOv10仅占~200MB，Depth需~500MB |
| Apple MPS | MPS | CPU | MPS与Depth Anything的某些算子不兼容 |
| 仅CPU | CPU | CPU | 无可选设备 |

这一策略的工程价值在于：系统无需用户手动配置设备分配，即可在不同硬件环境下自动选择最优运行模式，显著降低了部署门槛。同时，当GPU推理过程中发生运行时错误（如MPS的算子不支持），系统会自动回退至CPU重新执行当前帧的深度估计，确保不会因单帧错误导致整个系统崩溃。这种"优雅降级"的设计理念贯穿了整个系统的工程实现——在性能与稳定性之间，始终优先保证稳定性。

**深度图生成流程：**

1. 输入BGR图像 → 转换为RGB → 转换为PIL格式
2. 送入Depth Anything V2 pipeline → 输出PIL深度图
3. 转换为numpy数组 → 归一化到[0, 1]

$$d_{norm} = \frac{d - d_{min}}{d_{max} - d_{min}}$$

需要强调的是，Depth Anything V2输出的是**相对深度**而非绝对深度——深度值的大小关系是可靠的（近处深度值大、远处深度值小），但绝对尺度需要额外标定。这一特性直接决定了后续深度→距离映射策略的设计：系统采用线性映射而非直接使用深度值，正是因为相对深度的绝对值没有物理意义。

**区域深度提取策略**

深度估计模型输出的是全帧深度图，但3D重建只需要每个检测框内的深度值。直接取框内所有像素的均值会受边界噪声和背景深度泄漏的影响。本系统采用**中心50%区域中值提取**策略：

$$d_{target} = \text{median}(D[y_1':y_2', x_1':x_2'])$$

其中 $(x_1', y_1', x_2', y_2')$ 为检测框中心50%区域的坐标。取中值而非均值的原因是：中值对异常值（如边界处的深度跳变、遮挡物深度泄漏）更鲁棒。中心50%的裁剪比例是经验值——过大的裁剪比例会引入边界噪声（检测框边缘通常包含背景像素），过小则导致采样点不足，中值统计量不稳定。在实际测试中，50%的裁剪比例在精度和稳定性之间取得了最佳平衡。

**深度到距离的映射**

Depth Anything V2输出的是归一化相对深度 $d \in [0, 1]$，需要映射到物理距离。本系统采用线性映射：

$$Z = Z_{min} + d \times (Z_{max} - Z_{min}) = 1.0 + d \times 9.0 \quad (\text{m})$$

该映射假设深度估计的**相对排序**是准确的（近处深度值大、远处深度值小），仅绝对尺度需要标定。这一假设在Depth Anything V2的大量实验中已被验证成立 [[2]](https://arxiv.org/abs/2406.09414)。映射范围[1m, 10m]的选择基于城市驾驶场景的典型关注距离——1米以内通常已在2D检测框内不可见（目标过大，部分超出画面），10米以外则风险等级较低（城市道路限速下制动距离充足）。对于高速公路场景，可通过调整 $Z_{max}$ 参数将映射范围扩展至更远距离。

#### 4.2.3.2 3D重建实现

**三重约束的工程实现**

单目3D重建是经典的欠约束问题——从2D图像恢复3D信息，自由度大于约束数。本系统通过三重约束逐步收紧解空间，将无限解空间压缩为唯一解：

**约束一：针孔相机反投影（几何约束）**

已知2D像素坐标 $(u, v)$ 和深度 $Z_c$，利用相机内参矩阵 $K$ 反投影：

$$\begin{bmatrix} X_c \\ Y_c \\ Z_c \end{bmatrix} = Z_c \cdot K^{-1} \begin{bmatrix} u \\ v \\ 1 \end{bmatrix}$$

当缺乏标定数据时，系统自动估算内参：

$$K_{est} = \begin{bmatrix} W & 0 & W/2 \\ 0 & W & H/2 \\ 0 & 0 & 1 \end{bmatrix}$$

其中 $W, H$ 为图像宽高。该近似假设焦距等于图像宽度（对应约53°水平视场角），主点位于图像中心。这一近似在车载场景中是合理的——大多数车载摄像头的水平视场角在50°-60°之间，焦距近似等于图像宽度的误差在可接受范围内。对于需要更高精度的应用场景，系统支持通过配置文件加载标定后的相机内参矩阵，替换默认的估算值。

**约束二：先验尺寸约束（尺寸约束）**

单目视觉无法恢复绝对尺寸——同一辆车的2D投影在不同距离下尺寸不同，仅凭2D信息无法区分"近处小车"和"远处大车"。本系统使用类别先验尺寸约束3D框尺寸，如表4-7所示。

**表4-7 各类别先验3D尺寸（来源：KITTI数据集统计 [[5]](https://www.cvlibs.net/datasets/kitti/)）**

| 类别 | 高度(m) | 宽度(m) | 长度(m) | 数据来源 |
|------|--------|--------|--------|---------|
| car | 1.52 | 1.64 | 3.85 | KITTI Car均值 |
| truck/bus | 3.07 | 2.63 | 11.17 | KITTI Truck+Van均值 |
| motorcycle | 1.50 | 0.90 | 2.20 | KITTI Cyclist近似 |
| bicycle | 1.40 | 0.70 | 1.80 | KITTI Cyclist近似 |
| person | 1.75 | 0.60 | 0.60 | 标准成人尺寸 |

先验尺寸的引入使3D重建从"完全欠约束"变为"部分约束"——3D框的尺寸不再自由变化，而是被限制在类别先验值附近。这一约束的代价是丧失了对非标准尺寸车辆（如改装车、超宽货车）的精确建模能力，但在城市道路的常规交通场景中，先验尺寸的统计有效性足以保证合理的3D重建结果。系统在`BBox3DEstimator`类中维护了一个完整的先验尺寸字典`DEFAULT_DIMS`，不仅包含交通目标，还包含室内场景的常见物体（椅子、沙发、桌子等），为系统向室内场景扩展预留了接口。

**约束三：朝向估计（方向约束）**

朝向角 $\psi$ 通过几何启发式方法估计：

$$\psi = \alpha + \theta_{ray}$$

其中 $\theta_{ray} = \arctan(X/Z)$ 为相机到目标的射线角，$\alpha$ 根据2D框宽高比判断：

$$\alpha = \begin{cases} +\pi/2 & \text{宽高比} > 1.5 \text{ 且目标在图像左侧} \\ -\pi/2 & \text{宽高比} > 1.5 \text{ 且目标在图像右侧} \\ 0 & \text{其他（正面/背面视角）} \end{cases}$$

该启发式规则的物理依据是：当车辆侧面朝向相机时，2D框的宽高比显著增大（宽>高），且车辆朝向与其在图像中的位置相关——左侧的车辆通常朝右行驶，右侧的车辆通常朝左行驶。这一规则虽不能覆盖所有情况（如车辆正在转弯），但在直道场景中具有较高的准确性。宽高比阈值1.5的选择基于KITTI数据集的统计观察：侧面视角的车辆宽高比通常在1.5-2.5之间，正面/背面视角通常在0.5-1.2之间，1.5是一个合理的分界点。

#### 4.2.3.3 双重滤波平滑实现

深度估计的帧间波动是3D重建最严重的工程问题。同一像素在相邻帧的深度值差异经反投影放大后，可导致3D位置跳变1-2米。本系统采用双重滤波策略：

**第一重：卡尔曼滤波**

采用11维状态向量的卡尔曼滤波器：

$$\mathbf{x} = [x, y, z, w, h, l, \psi, \dot{x}, \dot{y}, \dot{z}, \dot{\psi}]^T$$

状态转移矩阵采用匀速运动模型：

$$F = \begin{bmatrix} I_7 & \Delta t \cdot I_4 \\ 0 & I_4 \end{bmatrix}$$

其中 $I_7$ 和 $I_4$ 分别为7阶和4阶单位矩阵，$\Delta t$ 为帧间隔。观测矩阵 $H$ 将11维状态映射到7维观测空间（位置3维+尺寸3维+朝向1维）：

$$H = \begin{bmatrix} I_7 & 0_{7 \times 4} \end{bmatrix}$$

卡尔曼滤波器的预测-更新循环遵循标准流程：预测步骤中，状态按匀速模型外推 $\hat{x} = F \cdot x_{t-1}$，协方差按 $\hat{P} = F \cdot P \cdot F^T + Q$ 传播；更新步骤中，计算卡尔曼增益 $K = \hat{P} \cdot H^T \cdot (H \cdot \hat{P} \cdot H^T + R)^{-1}$，然后用观测值修正预测值 $x = \hat{x} + K \cdot (z - H \cdot \hat{x})$。

**关键参数设计：** 观测噪声协方差 $R$ 中，深度方向 $R_{zz}=2.0$，远大于水平方向的 $R_{xx}=R_{yy}=0.1$。这一非对称设计的物理依据是：深度估计的噪声确实远大于2D检测的噪声，滤波器应在深度方向给予运动模型预测更高的权重。类似地，朝向角的观测噪声 $R_{\psi\psi}=0.3$ 也较高，因为朝向估计的启发式方法本身精度有限。

**表4-8 卡尔曼滤波器关键参数**

| 参数 | 值 | 物理含义 | 设计理由 |
|------|----|---------|---------|
| $R_{xx}, R_{yy}$ | 0.1 | 水平位置观测噪声 | 2D检测精度较高 |
| $R_{zz}$ | 2.0 | 深度方向观测噪声 | 深度估计噪声大，需强平滑 |
| $R_{\psi\psi}$ | 0.3 | 朝向角观测噪声 | 启发式朝向估计精度有限 |
| $Q_{vv_z}$ | 0.01 | 深度速度过程噪声 | 降低深度方向的预测不确定性 |
| $P_0$ | $I_{11}$ | 初始状态协方差 | 较大的初始不确定性，快速收敛 |

观测噪声协方差 $R$ 的非对称设计是本系统卡尔曼滤波器的核心创新点。标准卡尔曼滤波器通常使用各向同性的观测噪声（$R = \sigma^2 I$），这在所有维度噪声水平相近时是合理的。但在本系统中，深度方向的噪声远大于水平方向，若使用各向同性的 $R$，滤波器要么在深度方向平滑不足（$R$ 过小），要么在水平方向过度平滑（$R$ 过大）。非对称 $R$ 的设计使滤波器在深度方向更依赖运动模型预测（因观测不可靠），在水平方向更依赖观测值（因观测可靠），实现了维度自适应的平滑效果。

**第二重：指数移动平均（EMA）**

对卡尔曼滤波后的输出再做时间域平滑：

$$\hat{x}_t^{EMA} = \alpha \cdot x_t + (1-\alpha) \cdot \hat{x}_{t-1}^{EMA}$$

其中 $\alpha = 0.7$。该值的选择权衡了平滑性与延迟：

- $\alpha$ 过小（如0.3）→ 平滑性好但延迟大，120km/h下100ms延迟对应3.3m距离误差
- $\alpha$ 过大（如0.9）→ 延迟小但平滑性差，3D框仍有明显抖动
- $\alpha = 0.7$ → 在平滑性和延迟之间取得平衡

对于朝向角等周期性变量，EMA需要特别处理角度卷绕问题：当差值超过 $\pi$ 时需加减 $2\pi$，否则0°和359°的均值会被错误计算为179.5°而非-0.5°。系统在EMA步骤中实现了角度卷绕修正：

$$\Delta\psi = \psi_t - \hat{\psi}_{t-1}^{EMA}, \quad \Delta\psi \leftarrow \begin{cases} \Delta\psi - 2\pi & \Delta\psi > \pi \\ \Delta\psi + 2\pi & \Delta\psi < -\pi \\ \Delta\psi & \text{otherwise} \end{cases}$$

**双重滤波的效果：** 实测表明，双重滤波将3D位置的帧间抖动从1-2米降低到0.1-0.2米，减少80%以上。这一改善直接提升了速度估算的可靠性（速度噪声从108-216 km/h降至11-22 km/h），进而提升了风险场中速度拉伸效应的准确性。双重滤波的必要性在于：卡尔曼滤波虽能有效抑制高频噪声，但对低频漂移（深度估计的系统性偏差）效果有限；EMA则通过时间域的低通滤波特性补充了这一不足，两者互补形成了完整的平滑方案。缺少任何一重，3D位置的稳定性都将显著下降——仅用卡尔曼滤波时，低频漂移导致3D框缓慢"游走"；仅用EMA时，高频抖动虽被衰减但仍有残余振荡，影响可视化效果和速度估算精度。

**3D重建模块的输出与下游衔接**

经过三重约束重建和双重滤波平滑后，3D重建模块为每个跟踪目标输出完整的3D描述：

$$\text{3D\_output} = \{track\_id, (X, Y, Z), (W, H, L), \psi, (v_x, v_z)\}$$

这一输出是整条流水线中信息密度最高的环节——它将2D检测框+深度图这两个低级感知数据，融合为包含位置、尺寸、朝向、速度的完整3D描述。从数据流的角度看，3D重建模块是感知阶段与风险量化阶段之间的"信息瓶颈"——所有下游计算（风险场、SCF、BEV可视化）都依赖于这一输出的质量。因此，3D重建模块的精度和稳定性是整个系统性能的关键制约因素，也是工程优化投入最大的环节。

至此，感知检测阶段的三个核心模块（视频预处理、目标检测与跟踪、深度估计与3D重建）已完整实现。从原始视频帧到3D跟踪输出，数据经历了"2D检测→深度估计→3D重建→双重滤波→帧间关联"的完整转换链，每一步都为下一步提供了必要的前提条件。然而，感知检测仅回答了"周围有什么、在哪里、怎么动"的问题，尚未回答"有多危险"的问题。下一节将进入风险量化阶段，阐述如何将3D感知结果转化为可量化的风险评估。

### 4.2.4 轨迹预测模块

#### 功能定位

轨迹预测模块是连接"感知检测"与"风险量化"的桥梁模块。4.2.3节输出的3D位姿和速度描述了目标"当前在哪里、当前怎么动"，但风险量化不仅需要当前状态，还需要对未来短时间内的状态进行预判——一个正在匀速直行的车辆和一个正在变道的车辆，即使当前位置和速度相同，其风险水平也截然不同。轨迹预测模块的任务就是基于历史运动状态，推演目标在未来若干帧的可能位置，为风险量化提供前瞻性信息。

从信息论的角度看，轨迹预测是一个从"已知历史"到"未知未来"的推断过程，本质上是在不确定性下做决策。本系统采用**匀速运动模型**作为预测基线，这一选择基于以下工程考量：

1. **计算效率**：匀速模型仅需当前位置和速度即可预测，无需额外的模型参数或训练数据，计算开销可忽略
2. **短时有效性**：在预测时域较短（≤1秒）的条件下，匀速假设是合理的——大多数车辆在1秒内不会发生剧烈的加速度或转向变化
3. **鲁棒性**：复杂模型（如LSTM、Transformer）在训练数据不足时可能产生不可预测的外推行为，而匀速模型的外推行为是确定且可解释的

#### 匀速运动预测模型

对于每个已确认的跟踪目标，系统基于其当前3D位置和速度，预测未来 $N_{steps}$ 步的轨迹：

$$\mathbf{p}_{t+k} = \mathbf{p}_t + k \cdot \Delta t \cdot \mathbf{v}_t, \quad k = 1, 2, \ldots, N_{steps}$$

其中 $\mathbf{p}_t = (X_t, Z_t)$ 为当前3D位置（忽略Y轴高度），$\mathbf{v}_t = (v_{x,t}, v_{z,t})$ 为当前速度，$\Delta t$ 为帧间隔，$N_{steps}$ 为预测步数（默认10步）。

预测步数 $N_{steps}=10$ 的选择对应约0.33秒的预测时域（以30fps计算）。这一时域长度的设计权衡是：

- 过短（如3步≈0.1秒）→ 预测范围太近，风险预警的提前量不足
- 过长（如30步≈1.0秒）→ 匀速假设失效概率增大，预测误差急剧上升
- 10步≈0.33秒 → 在城市道路限速（60km/h≈16.7m/s）下，对应约5.5米的预测距离，足以覆盖紧急制动的反应时间

**预测不确定性建模**

匀速模型假设目标保持当前速度不变，但实际中目标可能加速、减速或转向。系统通过**线性增长的不确定性锥**来建模这一预测误差：

$$\sigma_{pred}(k) = \sigma_0 + k \cdot \sigma_{growth}$$

其中 $\sigma_0$ 为初始位置不确定性（取自卡尔曼滤波器的状态协方差），$\sigma_{growth}$ 为不确定性增长率（默认0.1m/步）。这一不确定性锥在BEV可视化中表现为轨迹线的渐变宽度——越远的预测点，线条越宽，表示不确定性越大。在风险量化中，不确定性锥影响风险场的扩散程度——预测位置的风险场覆盖范围随不确定性增大而扩展，体现了"越远越不确定，越不确定越需谨慎"的安全原则。

**轨迹预测与风险量化的衔接**

预测轨迹被传递至风险场引擎，用于计算**未来风险场**——将预测位置作为虚拟风险源，在预测位置处构建高斯风险场，其强度随预测步数递减：

$$I_{pred}(k) = I_0 \cdot e^{-\lambda k}$$

其中 $I_0$ 为当前风险场强度，$\lambda = 0.1$ 为衰减系数。衰减的物理含义是：预测越远的未来，风险场的不确定性越大，因此对当前风险决策的影响权重应越低。这一设计确保了系统在利用前瞻信息的同时，不会因远期预测的不确定性而过度预警。

**模块输出**

| 输出项 | 数据格式 | 后续消费者 | 关键用途 |
|--------|---------|-----------|---------|
| 预测轨迹点 | [(X, Z, σ), ...] × N_steps | BEV可视化 | 绘制预测轨迹渐隐线 |
| 预测风险场 | grid_h × grid_w | 风险场融合 | 与当前风险场加权融合 |
| 预测不确定性 | σ_pred(k) | 风险场扩散 | 控制预测风险场的空间扩展 |

轨迹预测模块虽然采用了最简单的匀速模型，但其输出为风险量化提供了不可或缺的前瞻性维度。在实际驾驶场景中，大多数危险情况并非由当前状态直接导致，而是由目标未来的运动趋势所引发——一辆正在变道的车辆，其当前风险可能不高，但预测轨迹将穿越自车行驶路径，因此需要提前预警。轨迹预测模块正是捕捉这一"趋势性风险"的关键环节。

### 4.2.5 风险分析与评估模块

#### 功能定位

本模块实现第3章"风险量化阶段"的全部功能，是系统从"感知"到"决策"的核心转换环节。模块接收3D轨迹和速度信息，输出风险场分布、SCF值和风险等级。如果说感知模块回答了"周围有什么、在哪里、怎么动"的问题，那么风险分析模块回答的就是"有多危险"的问题——这是一个从客观物理量到主观风险评估的映射过程。该映射的理论基础是驾驶风险场理论，其核心思想是将物理学中的"场"概念引入交通安全领域，用统一的数学框架描述各交通要素对驾驶安全的影响。

#### 4.2.5.1 风险场引擎实现

**网格化计算**

风险场在物理坐标系的网格上计算，网格参数如表4-9所示。

**表4-9 风险场网格参数**

| 参数 | 值 | 设计理由 |
|------|----|---------|
| 横向范围 | -8m ~ +8m | 覆盖左右各一个车道（标准车道宽3.75m） |
| 纵向范围 | 0m ~ 25m | 覆盖前方25米（城市道路典型关注距离） |
| 网格分辨率 | 0.1m | 平衡精度与计算量（40,000个网格点） |
| 网格尺寸 | 160 × 250 | 横向160列×纵向250行 |

网格初始化时，使用NumPy的`meshgrid`函数生成坐标矩阵：

$$X_{grid}, Z_{grid} = \text{meshgrid}(\text{linspace}(-8, 8, 160), \text{linspace}(0, 25, 250))$$

后续所有风险场计算均基于该网格进行向量化运算，避免了逐像素循环的低效实现。网格分辨率0.1m的选择是精度与计算量的折中——更细的分辨率（如0.05m）会使网格点数增加4倍至160,000个，计算时间相应增加，但SCF值的精度提升有限（因高斯函数本身是平滑的）。更粗的分辨率（如0.2m）虽可减少计算量，但在风险场等值线绘制时会出现明显的锯齿效应，影响可视化质量。

**高斯风险场构建**

对于每个风险源（运动车辆），构建旋转二维高斯风险场。该构建过程涉及三个关键步骤：速度拉伸、协方差旋转和马氏距离计算。

**步骤1：速度拉伸**

驾驶风险场理论的核心洞见是：运动物体的风险场不是各向同性的，而是沿运动方向被"拉伸"——速度越快，前方的影响范围越大。这一拉伸效应的物理解释是：运动车辆在行驶方向上具有动能，其前方区域比侧面区域更危险（正面碰撞的后果远比侧面擦碰严重）。系统通过调整纵向标准差实现拉伸：

$$\sigma_z^{eff} = \sigma_z \cdot w_{weather} + |\mathbf{v}| \cdot \alpha_{stretch}$$

其中 $\sigma_z = 2.0m$ 为基础纵向标准差，$w_{weather}$ 为天气因子（晴天1.0，雨天1.2，雪天1.5），$|\mathbf{v}| = \sqrt{v_x^2 + v_z^2}$ 为速度大小，$\alpha_{stretch} = 0.2$ 为拉伸系数。

天气因子 $w_{weather}$ 的设计考量是：恶劣天气条件下，制动距离增加、路面附着力下降，同一速度下的风险水平更高。因此，天气因子通过放大基础标准差来扩展风险场的覆盖范围——雨天时 $\sigma_z^{eff}$ 增加20%，雪天时增加50%，使得风险场的"警戒区域"更宽，预警更保守。拉伸系数 $\alpha_{stretch}=0.2$ 的物理含义是：每增加1m/s的速度，纵向标准差增加0.2m。以60km/h（16.7m/s）为例，速度拉伸贡献为 $16.7 \times 0.2 = 3.34m$，加上基础值2.0m，总纵向标准差为5.34m——这意味着风险场在车辆前方约5.34m处衰减到峰值的60.7%（高斯函数1σ处），在约10.68m处衰减到13.5%（2σ处），与直觉上"60km/h车辆前方约10米范围有风险"的认知一致。

**步骤2：协方差旋转**

速度方向不一定与坐标轴对齐（如车辆正在变道），因此需要将协方差矩阵旋转至速度方向。旋转角度由速度向量确定：

$$\theta = \arctan\left(\frac{v_x}{v_z}\right)$$

旋转后的全局协方差矩阵为：

$$\Sigma_{global} = R(\theta) \cdot \Sigma_{local} \cdot R(\theta)^T$$

其中旋转矩阵和局部协方差矩阵分别为：

$$R(\theta) = \begin{bmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{bmatrix}, \quad \Sigma_{local} = \begin{bmatrix} (\sigma_x^{eff})^2 & 0 \\ 0 & (\sigma_z^{eff})^2 \end{bmatrix}$$

协方差旋转的几何意义是：将一个沿坐标轴对齐的椭圆（局部坐标系中的等概率轮廓线）旋转至沿速度方向对齐。旋转后的椭圆长轴指向速度方向，短轴垂直于速度方向，准确反映了"运动方向风险更高"的物理事实。当速度为零时，$\theta=0$，旋转退化为恒等变换，风险场退化为各向同性的圆形高斯分布。

**步骤3：马氏距离计算与高斯函数**

在全局协方差矩阵下，计算网格上每个点到风险源中心的马氏距离：

$$d_M^2(\mathbf{p}) = (\mathbf{p} - \boldsymbol{\mu})^T \Sigma_{global}^{-1} (\mathbf{p} - \boldsymbol{\mu})$$

其中 $\boldsymbol{\mu} = (x_c, z_c)$ 为风险源中心位置。马氏距离是欧氏距离在协方差加权下的推广——它考虑了不同方向上风险衰减速率的差异（纵向衰减慢、横向衰减快），使得等风险轮廓线呈现为沿速度方向拉伸的椭圆，而非各向同性的圆。

最终的风险场值为：

$$f(\mathbf{p}) = \exp\left(-\frac{1}{2} d_M^2(\mathbf{p})\right)$$

**向量化计算优化**

马氏距离的计算在40,000个网格点上执行，若使用Python循环将极其缓慢。系统采用NumPy向量化计算：

1. 将网格坐标展平为 $N \times 2$ 的点矩阵：$\text{points} = \text{stack}[X_{grid}.ravel() - x_c, Z_{grid}.ravel() - z_c]$
2. 矩阵乘法计算加权距离：$\text{term1} = \text{points} @ \Sigma_{inv}$
3. 逐元素乘法+求和：$d_M^2 = \text{sum}(\text{term1} \times \text{points}, \text{axis}=1)$
4. 指数运算：$\text{grid\_vals} = \exp(-0.5 \times d_M^2)$
5. 重塑回网格：$\text{result} = \text{grid\_vals}.reshape(grid\_h, grid\_w)$

该向量化实现将单次风险场计算时间从约200ms（Python循环）降低到约2ms（NumPy向量化），加速100倍。这一优化是系统实现实时性的关键——以5个风险源计算，总耗时约10ms，仅占单帧处理时间的5%。

#### 4.2.5.2 SCF指标计算实现

**SCF（Safety Criticality Factor）** 是本系统提出的综合风险指标，将空间距离、相对速度、目标尺寸三个维度的风险信息融合为单一标量值。SCF的计算分为两步：

**步骤1：自车位置的风险场值提取**

自车位于坐标系原点 $(0, 0)$，其在风险场中的值为：

$$f_{ego} = \max_i f_i(0, 0)$$

即取所有风险源在自车位置产生的风险场值的最大值。取最大值而非求和的原因是：风险场值表示"危险程度"，多个中等风险源的叠加不应超过一个极高风险源的影响——一辆紧贴自车的高速车辆（$f \approx 0.8$）比三辆远处慢速车辆（$f \approx 0.3 \times 3 = 0.9$）更危险，但求和会得出相反的结论。取最大值确保了SCF反映的是"最紧迫的威胁"，而非"所有威胁的总和"。

**步骤2：SCF综合计算**

$$SCF = f_{ego} \times \left(1 + \frac{v_{rel}}{v_{ref}}\right) \times \left(1 + \frac{S_{target}}{S_{ref}}\right)$$

其中：

- $v_{rel} = \max(0, v_z^{target} - v_z^{ego})$ 为纵向相对速度（正值表示目标接近）
- $v_{ref} = 10$ m/s 为参考速度（归一化因子）
- $S_{target} = W \times L$ 为目标底面积
- $S_{ref} = 6.3$ m² 为参考面积（标准轿车底面积）

SCF公式的三项因子分别反映三个维度的风险贡献：

1. **空间因子** $f_{ego}$：反映自车在风险场中的暴露程度，取值[0, 1]
2. **速度因子** $(1 + v_{rel}/v_{ref})$：反映接近速度的威胁，接近速度越快风险越高
3. **尺寸因子** $(1 + S_{target}/S_{ref})$：反映目标尺寸的影响，大型车辆碰撞后果更严重

三项因子采用乘法而非加法融合，原因是三个维度之间存在**耦合放大效应**：一个近距离（$f_{ego}$高）、快速接近（$v_{rel}$大）、大型（$S_{target}$大）的目标，其风险远高于三个维度独立贡献的简单叠加。乘法融合自然地捕捉了这种耦合效应——当三个维度同时处于高水平时，SCF值呈指数级增长，触发最高级别的风险预警。

**SCF阈值与风险分级**

基于SCF值，系统将风险划分为四个等级，如表4-10所示。

**表4-10 SCF阈值与风险等级**

| 风险等级 | SCF范围 | 颜色编码 | 预警动作 | 物理含义 |
|---------|---------|---------|---------|---------|
| CLEAR | SCF < 0.3 | 绿色 | 无 | 周围无显著风险源 |
| LOW | 0.3 ≤ SCF < 0.6 | 黄色 | 提示 | 存在风险源但距离较远或速度较低 |
| MEDIUM | 0.6 ≤ SCF < 0.8 | 橙色 | 警告 | 风险源较近且有一定接近速度 |
| HIGH | SCF ≥ 0.8 | 红色 | 紧急警报 | 风险源极近或快速接近，需立即响应 |

阈值的设定遵循"宁早勿晚"的安全原则——0.3的CLEAR/LOW分界线意味着：只要风险场值超过0.3（约1.5σ距离处），系统即开始提示。这一保守策略可能导致一定的"过度预警"，但在主动安全领域，漏警（未预警但发生危险）的代价远高于虚警（预警但未发生危险）。阈值的具体数值经过多轮实车测试调优，在虚警率和漏警率之间取得了可接受的平衡。

#### 4.2.5.3 路面风险融合实现

除动态目标产生的风险场外，路面状况（湿滑、坑洼等）也会影响驾驶安全。系统通过路面检测模块识别路面状况，生成路面风险场，并与动态风险场融合。

**路面风险场模型**

路面风险场采用均匀分布模型：

$$f_{surface}(x, z) = w_{surface} \cdot \mathbb{1}_{road}(x, z)$$

其中 $w_{surface}$ 为路面风险权重（取决于路面状况），$\mathbb{1}_{road}$ 为路面区域指示函数。不同路面状况下的权重值如表4-11所示。

**表4-11 路面风险权重**

| 路面状况 | $w_{surface}$ | 制动距离倍数 | 来源 |
|---------|--------------|------------|------|
| 干燥 | 0.0 | 1.0× | 基准 |
| 潮湿 | 0.1 | 1.5× | 公安部交通管理局数据 |
| 积水 | 0.2 | 2.0× | 公安部交通管理局数据 |
| 结冰 | 0.3 | 3.0× | 公安部交通管理局数据 |

**动态+路面风险场融合**

动态风险场和路面风险场通过加权叠加融合：

$$f_{total}(x, z) = f_{dynamic}(x, z) + f_{surface}(x, z)$$

叠加而非取最大值的原因是：路面风险和动态目标风险是**独立且可叠加**的——一辆在结冰路面上快速接近的车辆，其风险水平应高于在干燥路面上同等速度接近的车辆，因为结冰路面上制动距离更长，可用的避险时间更短。叠加操作自然地反映了这种"风险叠加"效应。

融合后的总风险场在BEV鸟瞰图中以热力图形式可视化，颜色从蓝色（低风险）渐变至红色（高风险），为驾驶员提供直观的风险空间分布感知。热力图的颜色映射采用"jet"色图，其在低风险区域（蓝色）和高风险区域（红色）的视觉区分度最高，符合驾驶员对"蓝色=安全、红色=危险"的直觉认知。

## 4.3 本章小结

本章从工程实现的角度，详细阐述了第3章技术方案的具体实现过程。按照"数据输入与感知检测→风险量化→人机交互"三阶段的技术方案，逐一展开了各模块的软件设计与关键算法实现。全章的核心脉络可归纳为以下三个层次：

**第一层：感知检测（4.2.1-4.2.3节）**——解决"周围有什么、在哪里、怎么动"的问题。视频预处理模块将异构输入统一为标准格式；YOLOv10目标检测以无需NMS的高效推理实现逐帧检测；DeepSORT多目标跟踪通过外观+IoU+3D三重匹配建立帧间关联；Depth Anything V2深度估计从单目图像恢复深度信息；3D重建模块通过三重约束（几何+尺寸+方向）和双重滤波（卡尔曼+EMA）实现稳定的3D位姿估计。感知检测阶段的精度上限决定了整个系统的精度上限，因此获得了最高的工程优化优先级。

**第二层：风险量化（4.2.4-4.2.5节）**——解决"有多危险"的问题。轨迹预测模块基于匀速运动模型提供前瞻性信息；风险场引擎通过高斯风险场构建和向量化计算实现实时风险场计算；SCF指标将空间距离、相对速度、目标尺寸三维风险信息融合为单一标量；路面风险融合将环境因素纳入风险评估框架。风险量化阶段的核心创新是驾驶风险场理论的工程化实现——将物理学的"场"概念转化为可计算的数学模型，并通过向量化优化满足实时性要求。

**第三层：模块间衔接**——贯穿全章的"精度传递"与"信息增益"逻辑。每一模块的输出都是下一模块的输入，形成严格的数据依赖链：预处理的统一接口→检测的2D框→深度估计的深度图→3D重建的3D位姿→跟踪的速度和轨迹→风险场的SCF值。链路中任一环节的精度损失都会向下游传递并可能被放大（如深度误差经反投影放大为3D位置误差，再经差分放大为速度误差），因此系统优化遵循"上游优先"原则——感知阶段的精度提升对最终效果的边际收益最大。

本章的实现充分体现了"工程约束驱动设计决策"的核心思想：YOLOv10-S和Depth Anything V2-Small的选型受实时性约束驱动；双重滤波的设计受深度估计噪声约束驱动；三重匹配机制受同类目标区分约束驱动；SCF的乘法融合受风险耦合效应约束驱动。每一个技术决策都不是孤立的，而是在特定工程约束下的最优折中——这正是工程实现区别于理论研究的核心特征。
